<!-- Supabase carousels table schema — n8n HTTP Request node writes to this table -->

```sql
CREATE TABLE public.carousels (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  brand_id UUID NOT NULL REFERENCES public.brands(id),
  template_id UUID NOT NULL REFERENCES public.templates(id),
  image_style_id UUID NOT NULL REFERENCES public.image_styles(id),
  idea_text TEXT NOT NULL,
  post_body TEXT,              -- n8n writes this on completion
  status TEXT NOT NULL DEFAULT 'pending',  -- n8n updates to 'completed' or 'failed'
  slide_urls JSONB DEFAULT '[]',           -- n8n writes ImageBB URLs array here
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

HTTP Request node PATCH configuration:
```
Method: PATCH
URL: https://<PROJECT_REF>.supabase.co/rest/v1/carousels?id=eq.{{ $json.carousel_id }}
Authentication: Predefined Credential Type → Header Auth → "Supabase Service Role Key"
Additional Headers:
  Content-Type: application/json
  Prefer: return=representation
Body (JSON / Raw):
{
  "post_body": "{{ $json.post_body }}",
  "status": "completed",
  "slide_urls": {{ JSON.stringify($json.slide_urls) }},
  "updated_at": "{{ new Date().toISOString() }}"
}
```

n8n credential configuration:
```
Credential type: Header Auth
Name: Supabase Service Role Key
Header Name field: apikey
Header Value field: <service_role JWT from Supabase Dashboard → Settings → API>
```

CRITICAL: Do NOT use the native n8n Supabase node — it has a documented 403 bug (GitHub #17020) where it sends conflicting Authorization+apikey headers. The service_role key is rejected by Supabase's Kong gateway. Use the generic HTTP Request node only.

<task type="checkpoint:human-action">
  <name>Task 2: Create Supabase service role credential in n8n and replace Airtable nodes</name>
  <files>n8n Cloud workflow (no local files modified)</files>
  <action>
    Two-part task: (A) create the Supabase service role credential in n8n, then (B) replace all Airtable output nodes with HTTP Request nodes.

    PART A — Create the Header Auth credential:
    1. In n8n Cloud, open Credentials (left sidebar)
    2. Click "Add Credential" → search for and select "Header Auth"
    3. Fill in:
       - Credential Name: Supabase Service Role Key
       - Name (header name field): apikey
       - Value: paste the Supabase service_role JWT (Supabase Dashboard → Settings → API → service_role key, the long ~200-char JWT starting with eyJ)
    4. Save — the credential is now available to all workflow nodes in this n8n instance

    PART B — Replace Airtable output nodes:
    1. Open the carousel generation workflow in n8n Cloud editor
    2. Identify every Airtable node that writes generation results (look for nodes writing slide URLs, post body, or status)
    3. For each Airtable output node:
       a. Note what data it receives (inputs) and where it connects (outputs)
       b. Delete the Airtable node
       c. Add a new HTTP Request node in its place
       d. Configure (success/completed path):
          - Method: PATCH
          - URL: https://&lt;supabase-project-ref&gt;.supabase.co/rest/v1/carousels?id=eq.{{ $json.carousel_id }}
          - Authentication: Predefined Credential Type → Header Auth → Supabase Service Role Key
          - Send Headers: ON — add Content-Type: application/json and Prefer: return=representation
          - Body Content Type: JSON (Raw)
          - Body: { "post_body": "{{ $json.post_body }}", "status": "completed", "slide_urls": {{ JSON.stringify($json.slide_urls) }}, "updated_at": "{{ new Date().toISOString() }}" }
       e. If a failure branch exists, add a second HTTP Request node:
          - Same PATCH URL
          - Body: { "status": "failed", "updated_at": "{{ new Date().toISOString() }}" }
       f. Reconnect the node in the graph (same inputs/outputs as the deleted Airtable node)
    4. Verify no Airtable nodes remain in the output path
    5. Save the workflow

    DO NOT use the native n8n Supabase node (403 bug, GitHub #17020).
    DO NOT paste the service_role key directly in the header field — always reference the credential.
  </action>
