Create `supabase/migrations/20260306000008_billing_cron.sql`:

```sql
-- Enable pg_cron extension (safe to run if already enabled)
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Function: reset free-tier credits on the 1st of each month
-- SECURITY DEFINER runs with table owner privileges, bypassing RLS
-- Idempotency guard: only resets if last_reset_at is NOT in the current calendar month
CREATE OR REPLACE FUNCTION public.reset_free_tier_credits()
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER SET search_path = ''
AS $$
BEGIN
  UPDATE public.usage_tracking
  SET
    credits_remaining = credits_limit,
    last_reset_at = NOW(),
    updated_at = NOW()
  WHERE
    plan = 'free'
    AND date_trunc('month', last_reset_at) < date_trunc('month', NOW());
END;
$$;

-- Schedule: 00:00 UTC on the 1st of every month
-- cron.schedule is idempotent on the job name — safe to run multiple times
SELECT cron.schedule(
  'reset-free-credits-monthly',
  '0 0 1 * *',
  'SELECT public.reset_free_tier_credits()'
);
```

Key design decisions embedded in this SQL:
- `SECURITY DEFINER SET search_path = ''` — function runs with owner privileges (bypasses RLS on usage_tracking which is SELECT-only for authenticated users) and the empty search_path prevents search_path injection attacks.
- `date_trunc('month', last_reset_at) < date_trunc('month', NOW())` — the idempotency guard. If the cron job fires twice in January (e.g., after a retry), the second run finds `last_reset_at` already in January and skips. `last_reset_at` is initialized to the user's signup date by the `handle_new_user` trigger, which is always before the current month on first run.
- Only `plan = 'free'` rows are updated — Pro users have credits reset by the `invoice.paid` webhook in Plan 02.
- `updated_at = NOW()` — keeps the row's audit timestamp current.
